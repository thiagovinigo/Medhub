# US-023: Backend retorna erro estruturado com trace id para facilitar debug

**Feature:** FEAT-009
**Status:** Backlog
**Estimativa:** M
**Criado em:** 2026-06-06

---

## User Story
**Como** desenvolvedor
**Quero** que erros do backend retornem um ID de rastreamento (trace_id)
**Para** correlacionar erros reportados por usuarios com logs no Railway/Sentry

## Criterios de Aceitacao

```gherkin
Cenario 1: Erro 500 com trace_id
  DADO que uma excecao nao tratada ocorre no backend
  QUANDO o frontend recebe a resposta de erro
  ENTAO o payload contem: { "detail": "mensagem de erro", "trace_id": "uuid" }

Cenario 2: Trace_id aparece nos logs
  DADO que um erro ocorreu com trace_id "abc-123"
  QUANDO busco "abc-123" nos logs do Railway
  ENTAO encontro o stack trace completo do erro

Cenario 3: Logs em formato estruturado JSON
  DADO que o backend esta em producao
  ENTAO todos os logs (INFO, WARNING, ERROR) estao em formato JSON com campos: timestamp, level, message, trace_id
```

## Notas Tecnicas

- Substituir todos os `print()` por `logging.info/warning/error/exception`
- Configurar `logging.basicConfig` com formatter JSON (usar `python-json-logger`)
- Gerar `trace_id = uuid.uuid4()` no exception handler global e incluir no response e no log
- Integrar Sentry via `sentry_sdk.init(dsn=os.environ["SENTRY_DSN"])` com DSN em variavel de ambiente

## Dependencias

- Nenhuma dependencia de outras user stories.

## Criterios de Rejeicao

- DSN do Sentry hardcodado no codigo
- `print()` remanescentes no codigo de producao
- Trace_id diferente no log e na resposta para o mesmo erro
